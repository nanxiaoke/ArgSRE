# SRE 运营平台能力设计草案

## 1. 目标

SRE 运营平台是 ArgSRE 第一阶段的公共基础能力，目标是完成运营数据的自动化拉取、人工导入、解析、分析、展示、跟踪、审计和推送。

这个平台应服务于 SRE 的日常稳定性运营，而不是只做一次性的报表工具。它需要把分散在多个内部网页平台、工单系统、监控系统、变更系统和知识系统中的数据转成可持续跟踪的运营对象。

自动采集能力应作为后续各个模块的数据源底座。后续的告警治理、变更治理、容量治理、SLO 分析、故障复盘和风险跟踪等模块，都应尽量消费数据接入层输出的标准化数据，而不是直接绑定某个具体网页平台或采集脚本。

数据源需要作为平台级资源被统一管理，并允许各业务模块按自身场景进行编排配置。模块应声明自己需要什么数据对象、字段、时间窗口和质量要求，再由平台绑定一个或多个数据源实现。

结合既有“公有云高阶服务运维能力建设”讨论，运营平台还需要承载团队能力建设运营：围绕单个云服务的架构、部署基线、运行态、观测、应急、变更和重保 7 个维度，持续沉淀服务画像、能力评估、交付物、短板行动项和评审记录。详见 `docs/gemini-cloud-ops-discussion-notes.md` 和 `docs/sre-ops-capability-platform-design.md`。

## 2. 核心问题

当前关键场景是：大量需要采集的数据来自内部网页平台系统。这些系统通常有复杂的登录鉴权，但用户可以通过网页一次点击、等待自动鉴权、扫码、本机指纹认证或其他本机安全能力完成登录。

登录成功并获取 session 后，系统可以继续发送请求。因此平台需要具备：

- 无头或有头浏览器模拟登录能力。
- 人工操作录制能力。
- 前置操作条件记录能力，例如点击子页面、选择下拉框、设置时间范围、点击查询按钮。
- 页面请求记录能力。
- Session 复用与过期检测能力。
- 下次任务主动模拟访问和请求能力。
- 复杂响应结构的自动解析、字段抽取和数据组装能力。
- 自动配置与人工纠偏能力。
- 自动审计、异常发现和 IM 推送能力。
- 自动采集不可用时的人工导入能力。

## 3. 推荐模块

### 3.1 数据源管理

管理所有可采集的数据源，包括内部网页平台、接口、文件导出、工单系统、监控平台和其他内部服务。

每个数据源至少需要记录：

- 数据源名称和用途。
- 访问入口 URL。
- 鉴权方式。
- 采集方式：浏览器操作、请求回放、接口调用、文件导入等。
- 数据敏感级别。
- 采集频率。
- 负责人和告警接收人。
- 最近采集状态。
- 降级导入方式和导入模板。
- 可供哪些业务模块使用。
- 输出的数据模型和字段契约。
- 数据质量状态、可用性和最近更新时间。

数据源管理需要把“数据是什么”和“数据怎么来”解耦。一个数据源可以有多个接入方式，例如优先自动采集，失败后允许人工导入同一数据模型。

### 3.2 数据源编排

数据源编排用于让各业务模块以配置方式使用数据源，而不是在模块代码中硬编码采集逻辑。

第一阶段建议支持：

- 模块选择一个或多个数据源。
- 配置字段映射、过滤条件和时间范围。
- 配置主键、关联键和去重规则。
- 配置数据质量要求，例如更新时间、完整率、必填字段和失败容忍度。
- 配置降级策略，例如自动采集失败后切换为人工导入，或暂停分析并推送提醒。

后续可扩展为更完整的数据编排能力：

- 多数据源关联。
- 周期性聚合。
- 派生指标计算。
- 数据依赖 DAG。
- 模块级数据契约校验。

### 3.3 浏览器自动化采集器

负责处理复杂登录、页面操作、请求捕获和 session 复用。

建议分为两种模式：

- 录制模式：人工在受控浏览器中完成登录、点击、筛选、查询、导出等操作，平台记录关键页面行为、网络请求、必要参数和响应样例。
- 运行模式：平台按任务配置自动打开页面、等待鉴权状态、复用 session、重放关键请求或执行必要页面动作。

对于需要本机指纹认证、扫码或人工确认的登录流程，平台不应试图绕过安全机制，而应支持“人工完成登录后接管后续只读采集”。

典型认证流包括：访问目标 URL 后跳转统一认证页面；如果已有认证则自动跳回目标页面；如果短期认证丢失，则点击快速认证按钮后跳回目标页面；如果长期超时，则进入本机指纹认证流程，需要通过内部 IM 提醒用户人工完成认证。详见 `docs/data-source-auth-stack.md`。

### 3.4 人工导入

人工导入是自动采集不可用时的基础降级能力，也可以作为新数据源接入早期的低成本验证方式。

第一阶段建议支持：

- CSV 或 Excel 文件导入。
- JSON 文件导入。
- 从内部平台复制粘贴表格。
- 使用导入模板约束字段、类型和必填项。
- 导入前预览、字段映射和校验。
- 导入后生成审计记录，并标记数据来源为人工导入。

人工导入的数据应进入与自动采集相同的数据解析、标准化、分析和展示流程。后续模块不应关心数据来自自动采集还是人工导入，只应关心数据模型、时间范围、数据质量和来源可信度。

### 3.5 请求与会话管理

请求记录不仅要存 URL，还要存结构化上下文：

- 请求产生前的操作条件，例如菜单路径、按钮点击、下拉框选择、查询条件和等待条件。
- 请求方法、路径、查询参数和请求体结构。
- 必要请求头和 cookie 依赖。
- 鉴权状态和 session 生命周期。
- 请求之间的依赖关系。
- 分页、时间范围、筛选条件等动态参数。
- 响应结构摘要。

会话管理需要支持：

- Session 有效性检测。
- 自动续期或重新登录提示。
- 会话失效后的任务暂停与人工介入。
- Cookie、token 等敏感凭据的加密存储和权限隔离。

请求记录、操作条件、解析提取和模拟执行的详细设计见 `docs/request-recording-and-replay.md`。

### 3.6 数据解析与组装

内部平台响应的数据结构可能复杂，平台需要在自动解析和人工纠偏之间取得平衡。

能力建议：

- 自动识别 JSON、表格、嵌套结构、分页结构和列表详情结构。
- 支持字段映射、字段重命名、类型转换、单位转换和枚举归一。
- 支持跨请求组装，例如列表请求加详情请求、指标请求加元数据请求。
- 支持解析规则版本管理。
- 支持解析失败样例留存，但样例必须遵守内部安全要求。
- 支持可视化字段选择、解析预览、字段映射和人工确认。
- 支持把解析规则保存为数据源契约，供后续一键执行或业务模块触发。

### 3.7 自动配置与人工纠偏

这是平台的关键能力，不能只靠硬编码脚本。

推荐交互：

1. 用户录制一次完整采集流程。
2. 平台自动抽取登录状态、前置操作条件、关键请求、动态参数、响应结构和字段候选。
3. 用户在界面中确认必要操作、关键请求、字段含义、主键、时间字段、状态字段和异常判断规则。
4. 平台生成采集任务配置。
5. 运行中如果发现页面变化、字段缺失、解析失败或数据异常，平台进入纠偏流程。
6. 用户在失败样例或新页面上修正配置，平台生成新版本并保留审计记录。

### 3.8 分析、展示与跟踪

采集到的数据应转成可运营对象，而不是只展示原始表格。

第一阶段可支持：

- 趋势图、列表、聚合统计和状态看板。
- 异常项识别和待处理事项列表。
- 问题生命周期跟踪：发现、确认、处理、验证、关闭。
- 负责人、优先级、截止时间和处理记录。
- 历史变化和复发统计。

### 3.9 自动审计与问题推送

平台需要自动审计采集结果和运行状态，并在发现问题后及时推送。

审计对象包括：

- 数据采集是否成功。
- 数据是否缺失、突增、突降或长期不变。
- 关键指标是否超过阈值。
- 运营对象是否超期未处理。
- 数据源页面或接口结构是否变化。
- 采集任务是否频繁失败。

推送通道优先使用内部 IM 工具接口。推送内容应包含问题摘要、影响范围、建议动作、负责人和平台链接。

## 4. 初步架构

```text
内部网页平台 / 内部系统
        |
        v
浏览器自动化采集器 / 请求回放器 / 人工导入
        |
        v
原始响应缓存与采集审计
        |
        v
解析规则引擎 / 字段映射 / 数据组装
        |
        v
数据源目录 / 数据源编排 / 数据契约
        |
        v
运营数据模型 / 任务状态 / 问题跟踪
        |
        v
看板展示 / 分析规则 / IM 推送
```

## 5. 关键设计原则

- 尊重内部登录鉴权，不绕过安全机制。
- 采集优先只读，所有可能触发写操作的页面行为必须显式标记和审批。
- 浏览器自动化用于获取合法 session 和发现请求，不应成为不可审计的黑盒脚本。
- 人工导入是正式降级能力，不是临时绕路；它必须复用同一套数据模型、校验和审计链路。
- 数据源是平台级资源，需要被统一治理，并以配置方式被不同业务模块编排使用。
- 数据源采集、业务分析/展示和消息发送必须解耦：数据源只输出标准数据，业务模块只消费数据契约，消息通道通过独立适配器接收通用消息。
- 业务工作流可以编排多个数据源；单个数据源失败应隔离并显式标记，避免无必要地阻断其他数据源。
- 配置、解析规则、纠偏记录和推送记录都要版本化。
- 先把数据采集闭环做稳，再逐步加入智能分析。
- 自动审计和告警规则要优先做到简单、可解释、可行动，避免把第一阶段做成复杂脆弱的根因推断系统。
- 所有敏感凭据、cookie、token 和响应样例必须只保存在内部环境。

## 6. 待确认问题

- 首批内部网页平台有哪些？每个平台的登录方式分别是什么？
- 第一批运营数据对象是什么？例如告警、工单、变更、巡检、容量、SLO、故障复盘。
- 各业务模块使用数据源时，第一版需要支持哪些编排能力？
- 数据源的数据契约由谁维护？是平台管理员、模块负责人，还是数据源负责人？
- 人工导入首批需要支持哪些格式和模板？
- 人工导入的数据质量如何标记？是否需要人工确认或双人复核？
- 是否允许保存 cookie/token？如果允许，存储介质和加密要求是什么？
- 是否允许有头浏览器长期运行，还是必须使用无头浏览器？
- 内部 IM 接口的认证方式、消息格式、频率限制和回调能力是什么？
- 平台是否需要多用户、多租户、权限隔离和操作审批？
- 采集任务是否存在访问频率限制，避免对内部平台造成压力？

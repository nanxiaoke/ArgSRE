# 请求记录、操作条件与模拟执行设计草案

## 1. 问题定义

内部网页平台通常不是进入目标页面后立即发送目标数据请求。真实流程往往是：

1. 进入目标页面。
2. 点击进入某个子页面或功能入口。
3. 设置下拉框、按钮、日期范围、搜索条件、分页条件等。
4. 点击查询、刷新、导出或切换视图。
5. 页面才发送真正的数据请求。
6. 平台需要捕获请求、解析响应、提取字段，并把整个过程保存为后续可自动执行的数据源获取流程。

因此，数据源配置不能只记录一个 URL 或一个 HTTP 请求。它需要记录前置页面操作、操作条件、关键请求、请求参数来源、响应解析规则、字段映射、数据质量校验和自动执行策略。

## 2. 核心概念

### 2.1 操作条件

操作条件是触发目标请求之前必须完成的页面状态和用户操作。

典型操作条件：

- 点击导航菜单进入子页面。
- 点击 Tab 或二级菜单。
- 选择下拉框。
- 输入搜索关键字。
- 设置日期范围。
- 勾选复选框。
- 点击查询按钮。
- 等待某个页面元素出现。
- 等待某个接口返回。
- 翻页或加载更多。

操作条件需要被结构化记录，而不是只保存成视频或脚本。

### 2.2 关键请求

关键请求是用户完成操作条件后产生的、包含目标数据的请求。

平台需要从大量静态资源、埋点、权限、配置、字典和业务接口中识别候选关键请求，并允许人工确认。

### 2.3 提取规则

提取规则描述如何从响应中拿到业务字段。

第一阶段至少需要支持：

- JSON 路径选择。
- 表格数组选择。
- 嵌套对象展开。
- 字段重命名。
- 类型转换。
- 时间格式转换。
- 枚举映射。
- 空值处理。
- 主键和去重规则。

### 2.4 采集流程

采集流程是可重复执行的数据源获取单元：

```text
认证准备
  -> 进入目标页面
  -> 执行操作条件
  -> 捕获或重放关键请求
  -> 解析响应
  -> 校验数据质量
  -> 输出标准化数据
  -> 记录审计
```

后续业务模块应触发采集流程，而不是直接操作浏览器或请求接口。

## 3. 录制流程

### 3.1 操作录制

用户在受控浏览器里完成一次真实操作流程。

平台记录：

- 页面 URL 变化。
- 点击、输入、选择、勾选、等待等操作。
- 操作目标元素的多种定位信息。
- 操作前后的页面状态。
- 触发的请求和响应摘要。
- 操作与请求之间的时间和因果关系。

元素定位建议同时记录可访问性定位、CSS selector、XPath、元素文本、元素附近上下文、所属 iframe、页面 URL 或路由。

自动执行时优先使用稳定定位方式，失败时再尝试降级定位，并进入人工纠偏。

### 3.2 请求候选识别

录制结束后，平台自动列出候选请求。

筛选维度：

- 请求路径是否像业务接口。
- 响应类型是否为 JSON、表格、文件或结构化文本。
- 响应体是否包含列表、总数、分页、业务字段。
- 请求是否由最后几步操作触发。
- 请求耗时和响应大小是否异常。
- 是否排除静态资源、埋点、权限检查、菜单配置、字典接口。

用户需要在候选请求中选择一个或多个关键请求。

### 3.3 操作条件确认

平台根据关键请求反推其前置操作条件。

用户可以确认：

- 哪些操作是必要的。
- 哪些操作只是浏览或误点。
- 哪些参数需要在每次运行时动态输入。
- 哪些等待条件代表页面准备完成。
- 是否允许点击查询、导出等可能有副作用的按钮。

确认后，平台生成操作条件序列。

## 4. 解析提取设计

### 4.1 可视化解析

解析配置必须对用户友好，不能要求用户手写复杂脚本作为主要路径。

推荐交互：

1. 平台展示响应样例。
2. 自动识别候选数据列表。
3. 用户点击或选择数组节点作为记录集合。
4. 平台展示表格预览。
5. 用户勾选字段、重命名字段、设置类型。
6. 用户选择主键、时间字段、状态字段和负责人字段。
7. 平台保存解析规则并生成预览结果。

### 4.2 JSON 响应

JSON 响应建议使用 JSONPath 或类似路径表达式保存规则。

```text
record_path: $.data.items[*]
fields:
  id: $.id
  name: $.name
  status: $.state
  updated_at: $.updateTime
```

用户不需要手写这些表达式，但平台内部应保存为可审计、可版本化的结构。

### 4.3 表格和 HTML 响应

如果数据只存在页面表格中，需要支持：

- DOM 表格识别。
- 表头映射。
- 行列提取。
- 分页采集。
- 虚拟滚动加载。
- 复制粘贴式人工导入作为降级。

### 4.4 多请求组装

很多页面会先请求列表，再请求详情或字典。

第一阶段建议支持轻量组装：

- 主请求：产生记录列表。
- 详情请求：按记录 ID 补充字段。
- 字典请求：把状态码、类型码转换为可读名称。
- 元数据请求：补充业务线、负责人、服务名等维度。

每个组装步骤都需要有超时、失败策略和数据质量标记。

## 5. 模拟执行方式

### 5.1 浏览器模拟

适用于请求强依赖页面状态、前端加密参数、iframe、复杂路由或实时 token 的场景。

流程：

1. 启动专用 profile。
2. 访问入口 URL。
3. 检查认证状态。
4. 执行操作条件序列。
5. 等待关键请求出现。
6. 捕获响应。
7. 解析并入库。

### 5.2 请求回放

适用于认证状态稳定、参数可控、关键请求可以脱离页面直接发送的场景。

流程：

1. 从会话存储获取 cookie/token。
2. 根据配置生成请求参数。
3. 发送关键请求。
4. 解析响应。
5. 如果失败，降级到浏览器模拟或人工介入。

### 5.3 混合模式

推荐默认支持混合模式：

- 先用浏览器完成认证和前置操作。
- 捕获关键请求。
- 如果请求可回放，则后续优先请求回放。
- 如果请求回放失败，再回到浏览器模拟。

## 6. 一键自动化与模块触发

数据源配置完成后，需要形成一个可执行的采集流程。

触发方式：

- 用户在数据源页面点击一键执行。
- 定时任务自动执行。
- 业务模块按需触发。
- 数据质量异常后重试触发。
- 人工纠偏后验证触发。

执行结果：

- 成功：写入标准化数据，更新数据源状态。
- 需要认证：暂停任务，IM 提醒负责人。
- 解析失败：进入纠偏流程。
- 页面操作失败：记录失败步骤，进入纠偏流程。
- 数据质量异常：标记风险并推送。

## 7. 配置模型草案

```yaml
data_source:
  name: example_internal_page
  entry_url: https://internal.example/page
  auth_profile: sre-default
  flow:
    mode: hybrid
    steps:
      - type: goto
        url: "{{entry_url}}"
      - type: wait_for_auth
        target_url_pattern: "/target"
      - type: click
        selector_ref: nav.alerts
      - type: select
        selector_ref: filter.region
        value: "{{region}}"
      - type: click
        selector_ref: button.search
      - type: wait_for_response
        request_ref: alerts.list
  requests:
    alerts.list:
      method: POST
      url_pattern: "/api/alerts/search"
      dynamic_params:
        start_time: "{{time_window.start}}"
        end_time: "{{time_window.end}}"
  extract:
    record_path: "$.data.items[*]"
    fields:
      alert_id: "$.id"
      service: "$.serviceName"
      severity: "$.level"
      status: "$.status"
      updated_at: "$.updatedAt"
  quality:
    primary_key: alert_id
    required_fields:
      - alert_id
      - service
      - severity
    freshness_minutes: 30
```

## 8. 人工纠偏

人工纠偏需要覆盖三类问题：

- 操作条件失效：按钮变了、菜单位置变了、页面路由变了。
- 关键请求变化：URL、参数、分页、响应结构变化。
- 解析规则失效：字段改名、嵌套层级变化、返回空数据。

纠偏界面应提供：

- 失败步骤定位。
- 最近一次成功与失败对比。
- 新旧请求差异。
- 响应样例预览。
- 字段重新选择。
- 保存为新版本。
- 用新版本立即试跑。

## 9. 设计原则

- 请求记录必须和操作条件绑定。
- 操作录制的结果要能被人工编辑，而不是只能重录。
- 解析提取必须可视化、可预览、可版本化。
- 一键执行和模块触发应走同一个采集流程。
- 浏览器模拟与请求回放要能互相降级。
- 任何敏感请求头、cookie、token 和响应样例都只保存在内部环境。

# L1 基础运维能力与运营平台呈现设计

## 1. L1 定位

当前团队大部分能力还处在 L1 初级阶段，因此 L1 不能被设计成“低分标签”或“落后状态”，而应定义为最低安全运行能力。

L1 的核心目标是守住底线：

- 出事时知道谁负责。
- 知道服务在哪里、入口在哪里、现网大概长什么样。
- 能看懂最基础的告警和监控。
- 能快速收集必要上下文并升级给正确的人。
- 明确哪些动作不能做，避免误操作扩大影响。
- 能持续跟踪短板，不让风险长期隐身。

一句话：L1 不是要求团队独立解决复杂问题，而是要求团队做到“有事不乱、升级有效、信息完整、边界清楚、风险可见”。

## 2. L1 能力定义

### 2.1 服务识别与责任边界

团队至少需要知道：

- 服务名称、服务别名、服务类型。
- 业务重要性和影响范围。
- SRE 负责人。
- 研发负责人。
- 值班/升级群。
- 关键系统入口。
- 当前团队接管边界。

平台需要呈现：

- 服务卡片。
- 负责人和升级链路。
- 当前接管状态。
- 最近一次信息确认时间。
- 信息缺失项。

### 2.2 最小服务画像

L1 不要求完整白皮书，但必须有最小服务画像。

最小画像包括：

- 管理面和数据面的基本概念。
- 核心组件列表。
- 主要外部依赖。
- 部署区域或资源池。
- 基础监控入口。
- 基础日志入口。
- 基础控制台或运维入口。
- 常见问题升级路径。

平台需要呈现：

- 服务画像完整度。
- 核心组件是否已登记。
- 管理面/数据面是否已区分。
- 外部依赖是否有最小清单。
- 入口链接是否可用。

### 2.3 基础观测能力

L1 阶段最重要的是能发现“可能出事了”，并能判断影响是否在扩大。

团队至少需要掌握：

- 服务是否有基础监控入口。
- 哪些告警代表用户可能受影响。
- 哪些告警只是提醒或低优先级风险。
- 如何查看最近告警。
- 如何查看核心指标趋势。
- 是否存在基本拨测或黑盒检查。

平台需要呈现：

- 告警入口。
- 核心指标入口。
- 拨测入口。
- 最近告警摘要。
- 未关联 SOP 的告警数量。
- 高噪声告警或长期无人处理告警。

注意：L1 阶段不要追求复杂根因分析。告警规则应简单、可解释、可行动。

### 2.4 事故升级与上下文收集

L1 阶段不要求独立止血，但必须保证升级有效。

一次有效升级至少应包含：

- 发生时间。
- 影响服务。
- 影响区域。
- 影响用户或租户。
- 当前告警。
- 关键指标截图或链接。
- 最近变更。
- 初步现象描述。
- 已做过的只读检查。
- 当前升级对象。

平台需要提供：

- 一键生成升级摘要。
- IM 推送模板。
- 事故跟踪卡片。
- 最近变更自动关联。
- 告警和指标链接聚合。
- 升级记录和响应状态。

### 2.5 只读检查能力

L1 阶段允许团队执行低风险、只读、可审计的检查动作。

典型只读检查：

- 查看服务状态。
- 查看实例列表。
- 查看组件版本。
- 查看基础资源水位。
- 查看最近告警。
- 查看最近变更。
- 查看日志摘要。
- 查看拨测结果。

平台需要呈现：

- 只读检查清单。
- 每个检查项的入口。
- 最近一次检查结果。
- 检查失败原因。
- 是否需要人工补录。

原则：L1 默认不执行限流、重启、隔离、下线、扩缩容等高风险动作，除非已有明确授权 SOP。

### 2.6 操作禁区与升级条件

守住底线不仅是知道能做什么，也要知道不能做什么。

L1 必须明确：

- 禁止操作清单。
- 需要研发确认的操作。
- 需要双人复核的操作。
- 必须升级的告警类型。
- 必须升级的影响范围。
- 必须升级的变更异常。

平台需要呈现：

- 当前服务的操作边界。
- 高风险操作提示。
- 需要确认的升级条件。
- 相关负责人和确认链路。

### 2.7 文档与入口可用性

L1 的一个常见风险是“知道有文档，但找不到；找到了，但过期”。

平台需要跟踪：

- 架构说明入口。
- Runbook 入口。
- 监控大盘入口。
- 日志入口。
- 工单/变更入口。
- 研发支持群入口。
- 最近更新时间。
- 文档 Owner。

L1 达标不是要求文档完美，而是要求关键入口不丢、责任人明确、过期风险可见。

## 3. 七维度下的 L1 最小要求

### 3.1 架构设计与依赖拓扑

L1 最小要求：

- 知道管理面和数据面的基本区别。
- 有一张基础架构图或结构说明入口。
- 知道 3-5 个最关键组件。
- 知道最关键外部依赖。
- 知道架构问题应该升级给谁。

平台呈现：

- 架构入口是否存在。
- 管理面/数据面是否已标注。
- 核心组件登记数。
- 关键依赖登记数。
- 架构 Owner。

### 3.2 现网部署基线

L1 最小要求：

- 知道服务部署在哪些区域、资源池或集群。
- 能查到当前主要组件版本。
- 能查到基础部署入口。
- 知道部署信息的权威来源。

平台呈现：

- 部署区域清单。
- 组件版本信息是否存在。
- 最近同步时间。
- 数据来源。
- 缺失项。

### 3.3 运行态全景感知

L1 最小要求：

- 能看到基础资源水位。
- 能看到数据面实例数量或业务对象数量。
- 能看到是否有重点租户或重点资源。
- 能判断是否存在明显异常上升、下降或无数据。

平台呈现：

- 基础水位摘要。
- 实例/资源数量摘要。
- Top 资源消耗对象入口。
- 数据更新时间。
- 数据缺失告警。

### 3.4 观测体系建设

L1 最小要求：

- 知道告警从哪里看。
- 知道核心监控从哪里看。
- 能区分紧急告警和普通提醒。
- 能找到拨测或黑盒检查入口。
- 知道告警升级给谁。

平台呈现：

- 告警入口和最近告警。
- 核心监控入口。
- 拨测状态。
- 未绑定升级人的告警。
- 未绑定 SOP 的关键告警。

### 3.5 运维应急与修复

L1 最小要求：

- 有事故升级模板。
- 有基础信息收集清单。
- 知道哪些检查可以做。
- 知道哪些动作不能做。
- 有研发/SRE 协同链路。

平台呈现：

- 一键升级摘要。
- 只读检查清单。
- 禁止操作清单。
- 升级链路。
- 当前事件状态。

### 3.6 变更与演进管理

L1 最小要求：

- 能查到最近变更。
- 知道当前服务是否处于变更窗口。
- 知道变更异常应该升级给谁。
- 能在事故升级时带上最近变更上下文。

平台呈现：

- 最近变更列表。
- 当前变更窗口。
- 变更 Owner。
- 变更与告警的时间关联。

### 3.7 重保与高可用看护

L1 最小要求：

- 知道是否存在重点客户、重点租户或重点资源。
- 有重点对象清单入口。
- 重要时段知道联系谁。
- 能在事故中识别是否影响重点对象。

平台呈现：

- 重点对象清单。
- 重点对象当前状态。
- 重点对象关联告警。
- 重保负责人和升级链路。

## 4. L1 在运营平台的首页呈现

L1 阶段首页应是“底线工作台”，而不是复杂分析大盘。

推荐首页模块：

1. 服务底线状态

   展示每个服务是否具备最小画像、负责人、升级链路、观测入口、只读检查清单。

2. 今日风险

   展示当前告警、采集失败、数据缺失、文档过期、重点对象异常。

3. 一键升级

   选择服务后自动聚合告警、指标、最近变更、负责人和 IM 模板。

4. 信息缺口

   展示缺失的组件、依赖、入口、Owner、Runbook、重点对象信息。

5. 只读检查

   支持手动执行或查看只读检查结果，不提供高风险动作按钮。

6. 行动项

   把缺口转成明确的补齐任务，显示负责人、截止时间和状态。

## 5. L1 达标口径

建议不要用“评分”作为 L1 达标口径，而使用“底线清单”。

一个服务可以认为具备 L1 底线能力，需要满足：

- 有明确 SRE Owner 和研发 Owner。
- 有有效升级链路。
- 有最小服务画像。
- 有基础监控和告警入口。
- 有最近变更入口。
- 有只读检查清单。
- 有禁止操作或高风险操作边界。
- 有事故升级模板。
- 关键入口最近一次确认未过期。

如果这些不满足，就不是“能力差”，而是“底线信息缺失”，平台需要直接生成补齐行动项。

## 6. L1 到 L2 的过渡

L1 守住底线，L2 才开始强调独立初步定界和授权止血。

过渡路径：

1. 补齐最小画像。
2. 补齐观测入口和告警解释。
3. 建立只读检查清单。
4. 固化事故升级模板。
5. 梳理 3-5 个最高频故障场景。
6. 为这些场景补充 Runbook。
7. 明确哪些止血动作可以授权给 SRE 执行。
8. 通过演练验证升级、检查和协同链路。

平台需要把 L1 缺口自动转成 L2 准备事项，而不是让团队凭感觉推进。

# ArgSRE 内网测试交付规范

## 1. 目的

由于外部开发环境无法访问内网真实系统，每次交付不能只包含代码和简短命令，必须同时提供一套可重复、可判定、可脱敏反馈的测试包。

测试指导的目标是让内网验证人员能够明确回答：

- 测试的是哪个版本。
- 测试前需要准备什么。
- 每一步应该看到什么。
- 什么情况算通过、部分通过或失败。
- 失败后应该停止在哪里。
- 哪些证据可以保留在内网。
- 哪些信息可以脱敏反馈。
- 下一轮修复如何复现同一个问题。

## 2. 每次交付必须包含的内容

### 2.1 版本标识

- Commit SHA。
- 可选验证标签。
- 变更摘要。
- 影响范围。
- 已知限制。

### 2.2 环境要求

- 操作系统要求。
- Node、Python、Java 等运行时版本。
- 浏览器及版本。
- 依赖安装方式。
- 网络、权限和本机硬件要求。
- 是否需要有头浏览器、指纹设备或人工交互。

### 2.3 测试分层

每个版本至少区分：

1. 环境自检

   验证运行时、依赖、浏览器和目录权限。

2. 外部可复现测试

   使用模拟服务验证代码基础链路。

3. 内网真实环境测试

   只验证必须依赖真实认证、页面和请求的部分。

4. 回归测试

   验证新增功能没有破坏已经通过的能力。

### 2.4 测试用例

每个测试用例必须包含：

- 用例编号。
- 测试目的。
- 风险级别。
- 前置条件。
- 操作命令。
- 人工操作。
- 预期终端输出。
- 预期页面行为。
- 预期本地文件。
- 通过条件。
- 失败分类。
- 是否允许继续后续用例。
- 可反馈字段。

### 2.5 安全边界

必须明确：

- 哪些目录禁止外发。
- 哪些文件禁止提交。
- 哪些字段会自动脱敏。
- 哪些内容仍需人工检查。
- 测试是否包含写操作。
- 测试是否可能影响生产系统。
- 如何立即停止测试。

### 2.6 结果汇总

测试结束后必须形成统一结果：

- 环境自检结果。
- 每个用例的通过状态。
- 首个失败用例。
- 失败阶段和错误分类。
- 候选请求数量等非敏感统计。
- 下一步建议。

## 3. 测试状态定义

### PASS

所有必要预期结果均满足，且未违反安全边界。

### PARTIAL

主链路部分成功，但存在不阻塞后续分析的问题，例如：

- 能进入目标页面，但记录不到部分操作。
- 能捕获请求，但候选数量过多。
- 能识别记录节点，但部分字段结构不稳定。

### FAIL

主链路无法继续，例如：

- 环境自检失败。
- 浏览器无法启动。
- 认证无法完成。
- 目标页面无法进入。
- 没有捕获任何目标请求。
- 工具异常退出。

### BLOCKED

不是代码故障，而是缺少外部条件，例如：

- 无测试权限。
- 指纹设备不可用。
- 内部系统维护。
- 目标页面无数据。
- 依赖安装源不可达。

## 4. 停止规则

遇到以下情况必须立即停止，不继续尝试：

- 页面出现非预期写操作确认。
- 工具尝试提交、删除、扩容、重启或修改配置。
- 浏览器打开了错误的生产操作页面。
- 输出中出现未遮蔽的 Cookie、Authorization 或凭据。
- 运行目录准备被加入 Git。
- 测试可能对内部系统产生明显访问压力。

停止后只记录失败阶段和脱敏现象，不继续扩大测试范围。

## 5. 反馈最小化原则

外部开发需要的是“可复现问题的最小结构”，不是内网原始材料。

优先反馈：

- 用例编号。
- Commit SHA。
- PASS/PARTIAL/FAIL/BLOCKED。
- 错误分类。
- 脱敏错误摘要。
- 抽象路径。
- 字段名称和类型。
- 数量和布尔状态。

避免反馈：

- 完整日志。
- 完整请求。
- 完整响应。
- 页面截图。
- 内部 URL。
- 内部标识组合。

## 6. 交付检查表

外部开发推送版本前确认：

- [ ] 完整模拟测试已通过。
- [ ] 测试指南已随版本更新。
- [ ] Commit SHA 已记录。
- [ ] 依赖和环境要求已写明。
- [ ] 测试用例具有明确预期结果。
- [ ] 已知限制已写明。
- [ ] 失败分类已定义。
- [ ] 安全停止条件已定义。
- [ ] 反馈模板已更新。
- [ ] 运行数据和凭据已被 Git 忽略。

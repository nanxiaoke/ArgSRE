# 下一步：最小核心模型推进建议

## 1. 推荐方向

下一步不建议直接做完整的“云服务画像”，也不建议直接做完整的“七维度能力评估”。

更好的切入点是：先设计并打通 **L1 服务接管卡片**。

L1 服务接管卡片是一个最小组合模型，包含：

- 最小云服务画像。
- L1 底线清单。
- 数据源绑定。
- 风险与缺口。
- 短板行动项。
- 一键升级上下文。

这样做的好处：

- 符合当前团队大部分能力处于 L1 的现实。
- 不会过早进入复杂成熟度评估。
- 能快速支撑“守住底线，不出事”。
- 能自然指导数据源平台需要采集哪些数据。
- 能形成第一个可落地的运营页面。

该模型必须保持通用性。DWS、DMS、CCE、RDS、Redis、OBS 等不同高阶服务都应复用同一张通用接管卡片，再通过服务类型扩展字段补充各自特有对象和风险边界。

## 2. 为什么不是先做完整云服务画像

完整云服务画像很重要，但一开始容易做得过大：

- 架构、部署、运行态、观测、应急、变更、重保信息都很多。
- 很多字段短期内可能拿不到。
- 如果直接追求完整，会变成文档库或表单工程。

第一阶段应只保留 L1 必需字段：出事时能找到入口、找到人、找到上下文、知道边界。

## 3. 为什么不是先做完整七维度评估

七维度能力评估适合后续运营，但现在直接做容易带来几个风险：

- 团队会感受到“被打分”，偏离能力建设视角。
- 大量维度缺少证据，评估容易变成主观填写。
- 还没有数据源和服务画像支撑，评估结果难以驱动行动。

因此，七维度先作为信息组织框架，不作为复杂评分系统。

## 4. L1 服务接管卡片的数据模型

### 4.1 服务基本信息

- 服务 ID。
- 服务名称。
- 服务别名。
- 服务类型。
- 业务重要性。
- 服务状态。
- SRE Owner。
- 研发 Owner。
- 值班群或升级群。
- 最近确认时间。

### 4.2 最小服务画像

- 管理面说明。
- 数据面说明。
- 核心组件列表。
- 关键外部依赖。
- 部署区域或资源池。
- 基础监控入口。
- 基础告警入口。
- 日志入口。
- 变更入口。
- Runbook 入口。

### 4.3 L1 底线清单

- Owner 是否明确。
- 升级链路是否有效。
- 管理面/数据面是否已区分。
- 核心组件是否已登记。
- 关键依赖是否已登记。
- 基础监控入口是否存在。
- 告警入口是否存在。
- 最近变更入口是否存在。
- 只读检查清单是否存在。
- 禁止操作或高风险操作边界是否明确。
- 一键升级模板是否存在。

### 4.4 数据源绑定

- 监控数据源。
- 告警数据源。
- 变更数据源。
- 部署基线数据源。
- 运行态数据源。
- 人工导入模板。
- 数据源健康状态。
- 最近采集时间。

### 4.5 风险与缺口

- 缺口类型。
- 所属维度。
- 风险等级。
- 缺口描述。
- 影响说明。
- 推荐补齐动作。
- 是否阻塞 L1 底线。

### 4.6 短板行动项

- 行动项标题。
- 所属服务。
- 关联缺口。
- 负责人。
- 协作人。
- 截止时间。
- 当前状态。
- 验收标准。
- 证据链接。

### 4.7 一键升级上下文

- 当前服务。
- 当前告警。
- 关键指标链接。
- 最近变更。
- 影响区域。
- 影响租户或重点对象。
- 已执行只读检查。
- 当前升级对象。
- IM 消息模板。

## 5. 第一版页面建议

### 5.1 服务列表页

用于总览所有纳管服务。

重点展示：

- 服务名称。
- SRE Owner。
- 研发 Owner。
- L1 底线状态。
- 当前告警数。
- 数据源健康状态。
- 缺口数量。
- 逾期行动项。
- 最近确认时间。

### 5.2 服务接管卡片页

用于单服务 L1 底线运营。

建议分区：

- 顶部：服务基本信息、Owner、升级群、接管状态。
- 左侧：最小服务画像。
- 中部：L1 底线清单。
- 右侧：当前风险、告警、数据缺失、最近变更。
- 下方：只读检查、一键升级、行动项。

### 5.3 数据源绑定页

用于配置该服务的基础数据来源。

第一版不追求自动化全覆盖，允许自动采集、人工导入和手工链接共存。

### 5.4 缺口行动项页

用于跟踪 L1 补齐计划。

重点不是评分，而是“缺什么、谁补、什么时候补、怎么验收”。

## 6. 建议实施顺序

1. 选一个具体高阶云服务作为样例。
2. 填一张 L1 服务接管卡片。
3. 根据卡片反推需要哪些数据源。
4. 设计服务列表页和服务卡片页。
5. 把缺失字段自动生成行动项。
6. 支持一键升级摘要。
7. 再逐步扩展到七维度能力项和 L2 过渡。

## 7. 下一轮讨论建议

建议下一轮先围绕一个样例服务，手工填一版 L1 服务接管卡片。

如果暂时不方便使用真实服务，可以用一个抽象服务样例：

- 服务名：云数据库服务。
- 管理面：API、控制台、调度器、任务中心。
- 数据面：实例、存储节点、代理节点。
- 外部依赖：IAM、VPC、DNS、监控、工单、计费。

通过样例把字段、页面和数据源需求走一遍，再决定哪些字段进入第一版产品模型。

# 数据源认证与浏览器采集技术栈草案

## 1. 典型场景

用户在浏览器输入目标 URL 后，页面会跳转到统一认证页面。

认证状态可能有三种：

1. 已有有效认证

   认证页面显示认证成功，随后自动跳转到目标页面。

2. 短期认证丢失

   认证页面显示快速认证按钮。用户或自动化点击后，认证恢复并跳转到目标页面。

3. 长期超时

   页面弹出指纹认证流程，需要人工在本机指纹识别器上完成认证。认证成功后跳转到目标页面。平台可以通过内部 IM 推送提醒用户进行指纹认证。

这个场景的本质是“本机可信浏览器会话接管”，不是传统账号密码登录。因此平台应尊重内部认证机制，只在合法认证完成后复用会话并采集只读数据。

## 2. 推荐技术栈

### 2.1 浏览器自动化核心：Playwright

首选 Playwright，原因：

- 支持 Chromium、Firefox、WebKit，内部平台通常优先使用 Chromium 或 Edge。
- 支持有头和无头浏览器，适合人工认证和后台采集两种模式。
- 支持持久化浏览器上下文，可以把 cookie、localStorage 等会话信息保存在独立 profile 中。
- 支持保存和复用认证状态。
- 支持监听请求和响应，适合识别页面背后的真实接口。
- 支持 HAR 记录，方便分析请求链路和离线排查。

建议首期基于 Node.js + TypeScript + Playwright 实现浏览器采集 Worker。原因是 Playwright 的生态、示例和调试体验在 TypeScript 下最完整，也方便和前端管理界面共享类型定义。

### 2.2 浏览器选择：Chromium 或 Microsoft Edge

首选 Chromium/Edge，而不是系统默认用户浏览器 profile。

建议：

- 为 ArgSRE 创建专用浏览器 profile 目录。
- 不直接复用用户日常浏览器主 profile。
- 认证需要人工参与时使用有头模式。
- 认证状态有效、任务稳定后，可尝试无头模式运行。
- 如果内部认证强依赖本机安全能力、企业插件、证书或 Edge 策略，则优先支持 Edge channel 或企业指定浏览器。

### 2.3 后端服务

推荐后端采用以下之一：

- Python FastAPI：适合快速实现数据接入、解析、任务编排、人工导入、分析 API。
- Node.js NestJS：适合与 Playwright Worker 同语言栈，类型一致性好。

如果第一阶段重点是浏览器自动化采集，建议：

- 浏览器采集 Worker 使用 Node.js + TypeScript + Playwright。
- 平台 API 可先用 Node.js/NestJS 或 Python/FastAPI，后续根据团队熟悉度确定。

### 2.4 任务与调度

第一阶段不必过早引入复杂工作流引擎。建议先实现轻量任务模型：

- 数据源任务配置。
- 周期调度。
- 手动触发。
- 失败重试。
- 等待人工认证状态。
- 超时暂停。
- IM 提醒。
- 审计日志。

后续如果任务依赖变复杂，再引入 DAG 或工作流编排。

### 2.5 存储

建议至少分三类存储：

- 元数据数据库：数据源、任务、模块编排、解析规则、审计记录。
- 采集数据存储：标准化后的运营数据和问题跟踪数据。
- 安全会话存储：cookie、token、storageState、浏览器 profile、指纹认证后的会话状态。

安全会话存储必须加密、隔离权限、避免进入代码仓库或普通日志。

## 3. 认证流程设计

### 3.1 状态机

```text
开始访问目标 URL
        |
        v
跳转认证页面
        |
        v
识别认证状态
        |
        +-- 已认证成功 --> 等待目标页面 --> 捕获请求 --> 采集数据
        |
        +-- 快速认证 --> 点击快速认证按钮 --> 等待目标页面 --> 捕获请求 --> 采集数据
        |
        +-- 需要指纹 --> IM 提醒用户 --> 等待人工认证 --> 等待目标页面 --> 捕获请求 --> 采集数据
        |
        +-- 认证失败/超时 --> 标记任务暂停 --> 推送失败原因 --> 等待人工处理
```

### 3.2 关键检测点

平台需要配置或自动学习以下检测点：

- 认证页面 URL 模式。
- 目标页面 URL 模式。
- 认证成功页面特征。
- 快速认证按钮选择器。
- 指纹认证页面特征。
- 认证超时时间。
- 目标页面关键元素。
- 关键数据请求 URL 模式。

这些检测点应允许人工纠偏和版本化。

## 4. 采集运行模式

### 4.1 录制模式

用户打开受控浏览器，输入目标 URL，完成认证、点击、筛选、查询等操作。

平台记录：

- 页面跳转链路。
- 用户关键操作。
- 请求和响应摘要。
- 关键接口候选。
- 查询参数和动态参数。
- 响应结构样例。
- 认证状态变化。

### 4.2 运行模式

平台根据数据源配置自动访问目标 URL。

运行逻辑：

1. 使用专用 profile 或 storageState 启动浏览器。
2. 访问目标 URL。
3. 判断认证状态。
4. 已认证则继续。
5. 短期丢失则点击快速认证按钮。
6. 长期超时则通过 IM 提醒人工指纹认证。
7. 等待目标页面或关键接口响应。
8. 捕获并解析数据。
9. 写入标准化数据模型。
10. 记录审计日志。

## 5. IM 提醒设计

当任务进入“需要指纹认证”状态时，通过内部 IM 发送提醒。

提醒内容建议包含：

- 数据源名称。
- 任务名称。
- 认证原因：长期超时或需要本机指纹。
- 操作窗口链接。
- 超时时间。
- 任务负责人。

用户完成指纹认证后，平台继续等待目标页面跳转。如果超时仍未成功，则任务暂停并推送失败消息。

## 6. 风险与约束

- 不绕过指纹认证，不模拟或破解本机安全能力。
- 不保存用户指纹信息。
- 不复用用户日常浏览器主 profile。
- 不把 session 文件写入代码仓库。
- 不在普通日志中记录 cookie、token、完整请求头或敏感响应内容。
- 对可能触发写操作的按钮和请求做阻断或审批。
- 首期采集任务应默认只读。

## 7. 选型结论

首期推荐技术栈：

- 浏览器自动化：Playwright
- 语言：Node.js + TypeScript
- 浏览器：Chromium 或 Microsoft Edge 专用 profile
- 认证模式：有头浏览器完成认证，认证有效后可切换无头运行
- 会话管理：专用 profile + storageState，敏感内容加密保存
- 请求捕获：Playwright request/response 事件 + HAR
- 人工介入：平台任务状态 + 内部 IM 提醒
- 后端 API：Node.js/NestJS 或 Python/FastAPI，先按团队熟悉度选择
